# Space / Board — спецификация фичи (мудборд-холст внутри проекта)

> **Назначение этого файла.** Это полное ТЗ для реализации нового раздела **Space
> (доска / canvas)** — бесконечного холста внутри Проекта, куда можно кидать
> картинки из библиотеки, писать заметки-стикеры, добавлять текст и фигуры.
> Документ написан так, чтобы **новая AI-сессия могла начать реализацию с нуля**,
> ничего не додумывая. Сначала прочитай `CLAUDE.md`, затем этот файл.
>
> Источник дизайна — скриншоты референса GatherOS (4 шт., присланы владельцем).
> Ниже разобрано, как это должно выглядеть и работать в нашем нативном SwiftUI.

---

## 0. TL;DR

- **Что это:** бесконечный зумируемый холст (как Milanote / FigJam / tldraw),
  привязанный к **Проекту** (в коде — `spaceName`). На холст кладутся элементы:
  изображения из библиотеки, стикеры-заметки, текст, фигуры, фреймы.
- **Зачем:** визуально собрать мудборд проекта, а не только список карточек.
- **Главная сложность:** это **новая модель данных** (раскладка: координаты,
  размеры, z-порядок) + **кастомный canvas** на SwiftUI (пан/зум/выделение/ресайз).
- **Подход:** строго по фазам (см. §9), каждая фаза проверяется сборкой через CI.

---

## 1. Как это вписывается в текущее приложение

Сейчас «Проект» = просто значение `spaceName: String?` у ассетов (плоская
группировка, см. `CLAUDE.md` §2). Вкладка **Projects** (`spacesGrid` в
`GalleryGridView.swift`) показывает карточки проектов; клик по карточке сейчас
фильтрует библиотеку по этому проекту.

**Что меняем:**
- У карточки проекта появляется действие **«Open Board»** (открыть доску) —
  отдельный полноэкранный режим холста для этого проекта.
- Доска — это **отдельный визуальный слой поверх данных проекта**. Картинки на
  доске ссылаются на существующие `AssetItem` (по `id`); заметки/текст/фигуры —
  это новые объекты, живущие только на доске.
- `spaceName` остаётся как есть (группировка ассетов). Доска добавляется рядом,
  не ломая текущую модель.

**Важно:** не путать два зума.
- `uiZoom` (⌘+/−/0) — зум всего интерфейса приложения (уже есть).
- **Zoom холста** (− 100% + снизу справа) — это масштаб самого canvas, отдельная
  величина, живёт внутри Board-вью.

## 2. Модель данных (SwiftData)

Добавляем две новые `@Model`-сущности. Файл: `BoardModels.swift`.

### 2.1 `Board`
Метаданные доски. Одна доска на проект (пока 1:1; позже можно many).

```swift
@Model final class Board {
    @Attribute(.unique) var id: UUID
    var title: String            // "Untitled board" по умолчанию
    var projectName: String      // == AssetItem.spaceName, связывает доску с проектом
    var dateCreated: Date
    var dateModified: Date
    // Состояние вьюпорта, чтобы доска открывалась там же, где закрыли:
    var cameraX: Double          // пан холста
    var cameraY: Double
    var zoom: Double             // масштаб холста (1.0 = 100%)
    // Элементы доски (cascade delete — удаляем доску → удаляются её элементы)
    @Relationship(deleteRule: .cascade, inverse: \BoardItem.board)
    var items: [BoardItem] = []
}
```

### 2.2 `BoardItem`
Один элемент на холсте. Универсальный (тип в `kindRaw`), чтобы не плодить модели.

```swift
@Model final class BoardItem {
    @Attribute(.unique) var id: UUID
    var board: Board?

    var kindRaw: String          // "image" | "note" | "text" | "shape" | "frame"

    // Геометрия (в координатах холста, не экрана):
    var x: Double
    var y: Double
    var width: Double
    var height: Double
    var rotation: Double = 0     // градусы (на будущее)
    var zIndex: Int              // порядок наложения
    var isLocked: Bool = false

    // Для image: ссылка на ассет библиотеки (резолвим по id).
    var assetID: UUID? = nil

    // Для note / text:
    var text: String? = nil
    var fontName: String? = nil      // "Sans" → системный; на будущее кастомные
    var fontSize: Double? = nil      // 14 по умолчанию
    var isBold: Bool = false
    var isItalic: Bool = false
    var textAlignRaw: String? = nil  // "leading" | "center" | "trailing"
    var textColorHex: String? = nil  // цвет текста
    var fillColorHex: String? = nil  // фон стикера / заливка фигуры (жёлтый по умолч.)

    // Для shape:
    var shapeKindRaw: String? = nil  // "rectangle"|"ellipse"|"triangle"|"line"|"arrow"|"elbowArrow"

    // Для frame:
    var frameTitle: String? = nil
}
```

> **Почему ссылка на ассет по `assetID: UUID?`, а не SwiftData-relationship?**
> Чтобы удаление картинки из библиотеки не ломало доску жёстко: на доске можно
> показать «изображение недоступно», если ассет удалён. Резолв: один `@Query`
> ассетов в Board-вью + словарь `[UUID: AssetItem]`. (Альтернатива —
> `@Relationship var asset: AssetItem?`; решить на старте, но `assetID` проще и
> безопаснее.)

### 2.3 Enums (в том же файле)
`BoardItemKind`, `ShapeKind`, `TextAlign` — по аналогии с `AssetType`
(computed-обёртки над raw-строками, чтобы не падать на неизвестных значениях).

### 2.4 Регистрация схемы
В `manatherApp.swift` добавить новые модели в `Schema([AssetItem.self, Board.self, BoardItem.self])`.
⚠️ Это **миграция схемы** — но у нас уже сделано авто-восстановление стора
(`resetStore`), так что в худшем случае локальная БД пересоздастся, а не упадёт.
Для реальных пользователей позже понадобится lightweight migration; на этапе MVP
пересоздание допустимо.

## 4. Экран доски (BoardView) — общая раскладка

Соответствует **Скрину 1**.

```
┌─────────────────────────────────────────────────────────────┐
│  ‹назад      [ Untitled board ]              ▶ Present        │ ← топ-бар
├──┬──────────────────────────────────────────────────────────┤
│TB│                                                            │
│  │                  ⠿ точечная сетка                          │
│  │        [img]      [img]      [img]                         │ ← canvas
│  │                                                            │
│  │                                          − 100% +          │ ← зум холста
└──┴──────────────────────────────────────────────────────────┘
  TB = левая панель инструментов (плавающая, тёмная, скруглённая)
```

- **Топ-бар:** кнопка назад (закрыть доску → вернуться в проект); по центру —
  редактируемый заголовок доски (`Board.title`, тап → правка); справа — **Present**.
- **Фон:** тёмный (как viewer), точечная сетка (dot-grid), которая двигается и
  масштабируется вместе с камерой.
- **Зум холста:** нижний правый угол, «− 100% +», шаги ~10–25%, диапазон ~10–400%.

## 6. Холст (BoardCanvasView) — механика

Это ядро фичи. Чисто SwiftUI (без сторонних либ — правило проекта).

### 6.1 Камера и координаты
- Состояние: `camera = (x, y, zoom)` (хранится в `Board`, чтобы восстанавливалось).
- **Координаты холста** (canvas space) у каждого элемента (`BoardItem.x/y/...`).
- Перевод canvas → screen: `screen = (canvasPoint - cameraOffset) * zoom`.
- Рендер: `ZStack` элементов, каждый позиционируется `.position`/`.offset` с учётом
  камеры; общий контейнер масштабируется `scaleEffect(zoom)` ИЛИ (предпочтительно
  для чёткости) каждый элемент сам считает свой экранный rect. **Рекомендация:**
  применять `scaleEffect` ко всему слою элементов + `offset` для пана — проще, и
  для мудборда мягкость текста некритична (в отличие от UI-зума).

### 6.2 Жесты
- **Пан холста:** drag по пустому месту (инструмент Select) ИЛИ трекпад-скролл.
- **Зум холста:** `MagnificationGesture` (пинч) + кнопки «− %  +» + ⌘скролл.
  Зум **к точке под курсором** (anchor), не к центру.
- **Выделение:** клик по элементу. Клик по пустому — снять выделение.
- **Перемещение элемента:** drag выделенного.
- **Ресайз:** маркеры по углам/сторонам (см. Скрины 2,3 — точки по периметру).
- **Множественное выделение** (на будущее): рамкой-лассо. В MVP можно один объект.

### 6.3 Z-порядок и hit-testing
- Рисуем элементы в порядке `zIndex`.
- Hit-testing — сверху вниз (последний нарисованный ловит клик первым).
- Bring forward / send backward меняют `zIndex` (см. §7.1 тулбар выделения).

### 6.4 Undo/Redo
- Простой стек снимков (массив операций или снапшотов раскладки).
- На MVP можно ограничить: undo/redo для add/delete/move/resize.
- Хранить в `BoardViewModel`, не в БД.

### 6.5 Снапы (на будущее, не MVP)
- Привязка к соседним элементам / сетке, направляющие линии. В MVP — без снапов.

## 7. Типы элементов (BoardItemView)

### 7.1 Общий слой выделения + плавающая панель действий
Соответствует **Скрину 2** (выделенная картинка «SÈVE»):
- Вокруг выделенного — рамка + **круглые маркеры** по углам (ресайз).
- Сверху — **плавающая тёмная панель действий** (BoardSelectionToolbar) с кнопками
  (слева направо, по референсу):
  1. Duplicate (дублировать)
  2. Copy (копировать)
  3. Bring forward / поднять (стрелка вверх)
  4. Send backward / опустить (стрелка вниз)
  5. Lock (замок — заблокировать перемещение)
  6. Delete (корзина)
- Панель появляется над элементом и двигается вместе с ним.

### 7.2 Note (стикер-заметка)
Соответствует **Скрину 3**:
- **Жёлтый квадрат** (`fillColorHex` по умолчанию ~ `#FCEFA8`), мягкая тень.
- Внутри — редактируемый текст (`text`), двойной клик → режим правки.
- При выделении/правке сверху появляется **панель форматирования текста**
  (BoardTextToolbar, §7.6).
- Маркеры ресайза по углам.

### 7.3 Text (свободный текст)
- Как заметка, но **без фона** (прозрачный), только текст.
- Та же панель форматирования.

### 7.4 Shape (фигуры)
- rectangle, ellipse, triangle, line, arrow, elbowArrow.
- Заливка/обводка — `fillColorHex`. Для line/arrow — это «штрихи», рисуем `Path`.
- Создание: выбрал в флайауте → drag по холсту задаёт bbox.

### 7.5 Frame (фрейм)
- Прямоугольный контейнер с заголовком (`frameTitle`), для группировки секций
  мудборда. Элементы «внутри» фрейма двигаются вместе с ним (опционально на MVP —
  можно сделать фрейм просто визуальной рамкой без захвата детей).

### 7.6 Панель форматирования текста (BoardTextToolbar)
Соответствует **Скринам 3 и 4** (верхняя плашка):
- Шрифт: dropdown «Sans ▾» (на MVP — системный; список на будущее).
- Размер: поле «14» со степперами (▲▼).
- **B** (bold), **I** (italic).
- Выравнивание: leading / center / trailing.
- Цвет текста: «A ▾» с палитрой.
- Цвет заметки/заливки: круглый swatch (жёлтый) ▾ с палитрой.
- Появляется сверху по центру при выделении note/text.

## 10. Present mode и Export

- **Present** (кнопка в топ-баре): полноэкранный показ доски без тулбаров и сетки,
  возможно по «слайдам» = фреймам (стрелки ←/→ между фреймами). На MVP — просто
  чистый полноэкранный вид холста с навигацией зумом, без редактирования.
- **Export** (иконка download в левом тулбаре): варианты —
  (а) PNG-снимок холста (рендер через `ImageRenderer`),
  (б) интеграция с существующим **Context Pack** (доска → часть `CONTEXT.md`).
  На MVP — PNG-снимок.

## 11. Технические риски и решения

- **Кастомный canvas в SwiftUI — самое трудоёмкое.** Делать инкрементально (§9).
  Координаты/жесты/ресайз тестировать в Xcode на Маке (без Мака CI проверит лишь
  компиляцию, не поведение — это ОК для каркаса, но финальную отладку жестов
  делать на устройстве).
- **Миграция схемы SwiftData.** На MVP полагаемся на `resetStore` (БД может
  пересоздаться). Перед релизом — lightweight migration или версионирование.
- **Производительность.** На доске — только кешированные тамбнейлы
  (`CachedImageView`), как в сетке. Не декодировать оригиналы.
- **Не ломать существующее.** `spaceName`-группировка и вкладка Projects остаются;
  доска добавляется рядом. Не трогать `CachedImageView.loadedPath`, цветовую
  индексацию и пр.
- **Чистая изоляция (MainActor).** Новый код — на главном акторе (как весь UI);
  тяжёлое (рендер экспорта) — аккуратно, по образцу `FileManagerHelper`.
